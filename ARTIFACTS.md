# Артефакты системы

Артефакты — это контракты между ролями. Через них передаётся информация, и founder может их проверять на контрольных точках.

## Список артефактов

### 1. Product Requirements Document (PRD)
**От:** Product Manager
**Кому:** Analyst, Architect, Designer

**Входные данные:**
- Визия founder'а в любой форме (текст, диалог)
- Уточняющие вопросы PM → ответы founder'а

**Формат и хранение:**
- Актуальная версия: `docs/prd.md`
- Архив предыдущих версий: `docs/archive/prd/prd-v{N}-{YYYY-MM-DD}.md`
- Синхронизация с GitHub репозиторием

**Структура документа:**
- Цель продукта
- Целевая аудитория (персоны)
- Функциональные требования (фичи)
- Нефункциональные требования
- Приоритеты (MoSCoW)
- Критерии успеха
- Риски и ограничения
- Changelog (в конце документа)

**Версионирование:**
- Актуальная версия всегда в `docs/prd.md`
- При значимых изменениях — архивирование старой версии
- Changelog фиксирует: дату, автора (агента), суть изменений

**Шаблон:** `templates/prd.md`

### 2. Бэклог (GitHub Issues + Projects)
**От:** Product Manager
**Кому:** Analyst

**Формат:** GitHub Issues + GitHub Projects

**Структура:**
- Фичи → Задачи (без эпиков и stories)

**GitHub Projects:**

Колонки:
- Backlog → Ready → In Progress → Review → Done

Поля:
- Status (колонка)
- Priority (P0-P3)
- Assignee (роль-исполнитель)

**Лейблы:**

Тип:
- `feature` — новая функциональность
- `bug` — исправление ошибок
- `docs` — документация

Приоритет:
- `P0` — критично, блокирует релиз
- `P1` — важно, должно быть в релизе
- `P2` — средний приоритет
- `P3` — низкий приоритет

Роль:
- `pm`, `analyst`, `architect`, `designer`, `developer`, `qa`, `tech-writer`

**Связь с PRD:**
- Каждая фича/задача содержит ссылку на пункт PRD
- PM отслеживает покрытие PRD задачами
- При неполном покрытии PM обращается к founder'у

**Передача задач:**
- Assignee указывает текущую роль-исполнитель
- При завершении этапа роль меняет assignee на следующую роль
- Founder видит кто над чем работает

---

### 3. Спецификация задачи
**От:** Analyst
**Кому:** Developer, QA

**Формат:** GitHub Issue с шаблоном

**Глубина декомпозиции:**
- Одна задача = один Pull Request
- В среднем 2-3 задачи на фичу

**Содержание:**
- Описание задачи
- Критерии приёмки (Acceptance Criteria)
- Технические детали
- Зависимости
- Оценка сложности (S/M/L/XL)
- Ссылка на схему архитектора (если применимо)

**Связи:**
- Задача → ссылается на Фичу из бэклога
- Фича → ссылается на пункт PRD

**Оценка сложности:**
- Оценивает Developer
- Шкала: S (малая), M (средняя), L (большая), XL (очень большая)

### 4. Architecture Decision Record (ADR)
**От:** Architect
**Кому:** Developer, QA, Analyst

**Когда создаётся:**
- На старте проекта — общая архитектура и ключевые решения
- При новых фичах/требованиях — если они влияют на архитектуру
- Триггерит: Architect

**Хранение:**
- Общая архитектура: `docs/architecture/ADR.md`
- Частные ADR: `docs/architecture/adr/ADR-{N}-{title}.md`

**Содержание:**
- Context — контекст решения
- Problem — проблема/вызов
- Options — рассмотренные варианты
- Decision — принятое решение
- Consequences — последствия

**Связь с задачами:**
- Задачи ссылаются на соответствующий ADR
- ADR используется Developer и Analyst при работе
- ADR не порождает задачи напрямую

### 5. System Design
**От:** Architect
**Кому:** Developer

**Уровни детализации:**
- C4 модель: Context → Containers → Components
- Code-уровень не нужен — проектирует Developer при написании кода

**Диаграммы:**
- Формат: Mermaid в Markdown
- Типы: ER-диаграммы, API-схемы

**OpenAPI спецификация:**
- Формат: OpenAPI 3.0
- Хранение: `docs/api/openapi.yaml`

**Связь с ADR:**
- System Design ссылается на ADR
- Является графической иллюстрацией архитектурных решений

**Обновление:**
- Живой документ
- Должен быть консистентен с текущей системой
- Не должен противоречить реальному состоянию

---

### 6. Дизайн-система
**От:** Designer
**Кому:** Developer

**Подход:** Code-first

**Инструменты:**
- UI-kit: shadcn/ui / Chakra UI / Material UI (выбирается на старте)
- Styling: Tailwind CSS
- Компоненты: React-компоненты в коде

**Состав дизайн-системы:**
- Токены: цвета, типографика, отступы, тени — `src/design-system/tokens.css`
- Компоненты: кнопки, инпуты, модалки — `src/design-system/components/`
- Иконки: Lucide / Heroicons
- Документация: `docs/design-system.md`

**Хранение:**
- Всё в репозитории проекта
- Синхронизация с GitHub

**Workflow:**
1. Designer создаёт/редактирует токены и компоненты в коде
2. Developer использует готовые компоненты
3. Изменения дизайна = изменения в коде (PR)

**Версионирование:**
- Через Git (commits, PR)
- Changelog дизайн-системы в `docs/design-system.md`

### 7. UI-макеты
**От:** Designer
**Кому:** Developer

**Формат:**
- Markdown-описания экранов: `docs/ui-screens/`
- Прототипы в коде (опционально): генерируются на основе описаний

**Содержание описания экрана:**
- Название экрана
- Расположение блоков
- Навигация (откуда/куда)
- Используемые компоненты из дизайн-системы

**Связь с дизайн-системой:**
- Макеты ссылаются на компоненты из `src/design-system/`

**User Flows:**
- Формат: Markdown
- Хранение: `docs/user-flows/`

### 8. Pull Request
**От:** Developer
**Кому:** Developer (reviewer), QA

**Шаблон PR:**
- Ссылка на задачу
- Описание изменений
- Скриншоты (если UI)
- Чек-лист для ревью

**Code Review:**
- Проводит другой Developer на **другой LLM модели**
- Developer пишет юнит-тесты для своего кода

**Размер PR:**
- Лимит: до 1000 строк
- Если больше — дробить на несколько PR
- Одна задача может иметь несколько PR

**Мёрдж:**
- Мерджит Developer после получения approve
- Если замечания — исправить и выложить новый коммит
- Требования: approve + пройденные CI-проверки

**Связь с задачей:**
- PR **не закрывает** задачу автоматически
- Задача закрывается после успешного тестирования QA
- PR ссылается на задачу (но не закрывает)

### 9. Test Cases
**От:** QA
**Кому:** Developer, Founder

**Разделение ответственности:**
- **Unit-тесты:** пишет Developer, хранятся вместе с кодом
- **Интеграционные/E2E:** пишет QA, хранятся в Markdown

**Хранение:**
- Unit-тесты: рядом с кодом, фреймворк зависит от языка
- E2E/Интеграционные: `tests/cases/` в Markdown

**Содержание тест-кейса:**
- ID тест-кейса
- Ссылка на задачу
- Ссылка на Acceptance Criteria
- Описание
- Предусловия
- Шаги
- Ожидаемый результат
- Статус: Draft → Active → Deprecated

**Лог прогона:**
- Для каждого теста: Passed / Failed
- Комментарий при Failed

**Наборы тестов:**
- **Smoke:** основная функциональность, ~20% от всех тестов
- **Regression:** полный комплект для комплексного тестирования

---

### 10. Баг-репорт
**От:** QA
**Кому:** Developer

**Формат:** GitHub Issue с label `bug`

**Приоритет:**
- P0 — критический, блокирует работу
- P1 — важный, нужно исправить скоро
- P2 — средний приоритет
- P3 — minor, косметический
- Определяет: QA

**Содержание:**
- Описание бага
- Шаги воспроизведения
- Ожидаемое поведение
- Фактическое поведение
- Скриншоты/логи
- Окружение (версия, браузер, устройство)
- Ссылка на тест-кейс

**Жизненный цикл:**
- Open → In Progress → Fixed → Verified → Closed
- Верифицирует: QA

## Потоки артефактов

```
Founder Vision
      ↓
[PRD] → Product Manager
      ↓
[Backlog] → GitHub Issues
      ↓
[Task Spec] → Analyst
      ↓
[ADR + System Design] → Architect
[Design System + UI] → Designer
      ↓
[Pull Request] → Developer
      ↓
[Test Cases + Bugs] → QA
      ↓
[Docs] → Tech Writer
      ↓
Release
```
